设计总结 & 架构方案

✅ 已确认的设计决策

左栏 任务列表（进行中 / 待审查），完全替换原有项目树和资产面板
中栏 当前选中任务的独立 Agent 对话（单任务视图）
右栏 画布（不变）
任务创建 中栏空状态输入框，用户发消息即创建任务
Agent 行为 直接执行 tool → 完成后任务自动变为「待审查」
任务标题 Agent 根据对话内容自动生成

技术架构：3 个方案

A

全新 TaskStore + 内嵌消息

新建 taskStore，每个 task 包含自己的 messages[]。中栏直接渲染当前 task 的消息。聊天 SSE 逻辑从 taskStore 驱动

优点

  • 干净分离，不影响现有 chatStore
  • 每个 task 是独立单元

缺点

  • SSE 连接管理变复杂（每个 task 一个 session?）
  • 需要新建较多代码
B

改造 chatStore 为多线程

将现有 chatStore 扩展为支持多 thread。threadId 对应 taskId。中栏按 activeThreadId 切换显示

优点

  • 复用现有聊天 UI 组件
  • SSE 逻辑可以复用

缺点

  • chatStore 改造成本高
  • 旧单线程逻辑需要迁移
C

轻量 TaskStore + 独立 AgentSession

taskStore 只管任务元数据（id, title, status, sessionId）。中栏直接对接 Pandaria session API，每个任务 = 一个 Pandaria session。消息和 SSE 都由 session 驱动

优点

  • 消息存储交给 Pandaria，前端只管展示
  • 架构最干净，任务 = session

缺点

  • 依赖 Pandaria 的消息持久化（需要确认是否支持）
  • 离线不可用

推荐 A（本地优先，不依赖 Pandaria 消息持久化）。可以后续平滑迁移到 C。
